On-Device Architecture

🎙️ Audio │ Whisper-small.en (speech → text) ▼ 📄 Transcript ──▶ MiniLM (text → 384-dim vector) │ │ │ ▼ │ 🗄️ LOCAL VECTOR DB (sqlite-vec) │ cosine search of past patient notes │ │ ▼ ▼ 🧠 Llama-3.2-1B ◀── transcript + retrieved history │ ▼ 📋 JSON SOAP Note

All models quantized to INT8, exported with ExecuTorch, and run on the Snapdragon NPU.

(1:15) Architecturally: audio goes into Whisper for transcription; the transcript is embedded by MiniLM into a 384-dim vector; we cosine-search a local sqlite-vec database for relevant past notes; then Llama-3.2-1B takes the transcript plus that history and writes the SOAP note as JSON. Every model is INT8-quantized, exported with ExecuTorch, and runs on the Snapdragon NPU.

Why Vector Search?

  • An AI model turns a sentence into 384 numbers (an embedding).
  • Similar meaning → similar numbers.
  • "Find relevant notes" becomes "find the closest vectors" — by meaning, not keywords.

Cosine distance measures the angle between vectors:

0.0 = same meaning … 1.0 = unrelated

(2:25) Quick why-vector-search: an embedding model turns a sentence into 384 numbers, and similar meaning gives similar numbers. So "find relevant notes" becomes "find the closest vectors" — matched by meaning, not keywords. We measure with cosine distance: 0 is identical meaning, 1 is unrelated.

The Database Design

CREATE VIRTUAL TABLE patient_vectors USING vec0(
    patient_id   INTEGER PARTITION KEY,   -- search scoped per patient
    encounter_id INTEGER,                 -- links a hit to its visit
    embedding    FLOAT[384] distance_metric=cosine,
    +chunk_text  TEXT                     -- original note, returned with the hit
);
  • Plain SQL tables for patients, encounters, vitals
  • Partition key → fast and keeps patients isolated (privacy)
(2:50) Here's the core schema — a vec0 virtual table holding 384-dim cosine embeddings. Two things to notice: chunk_text returns the original note alongside the hit, and patient_id is a partition key — so search is fast AND one patient's data is physically isolated from another's. Plain SQL tables back the patients, encounters, and vitals.

Engineering Highlights

  • Self-contained & offline: vendored sqlite-vec + SQLite, statically linked → no runtime downloads.

  • Tested at every level: C unit test, Python suite, 5k-vector stress, on-device run.
